The Problem

Why NutriGuard?

Millions track meals but lack actionable, personalized guidance that considers their unique health conditions, deficiencies, and goals.

⚠️
Blind Spots in Nutrition
People unknowingly combine foods that block nutrient absorption — like drinking tea right after taking an iron supplement, reducing absorption by up to 60%.
📊
Generic Advice Doesn't Work
Existing apps offer one-size-fits-all guidance. A vegetarian with iron deficiency needs fundamentally different advice than someone tracking macros for muscle gain.
🕐
Timing Matters
Nutrient interactions depend on meal timing across the whole day. No app analyzes your daily meal timeline to detect cross-meal risks and patterns.
The Solution

Profile-Aware, AI-Powered Nutrition

NutriGuard lets users define their health goals, deficiencies, dietary preferences, supplements, and conditions — then uses AI agents to analyze every meal in that context.

What Users Can Define

  • Health Goals — fat loss, muscle gain, maintenance, energy, reduce deficiency, improve lab markers
  • Diet Type — vegetarian, eggetarian, vegan, non-vegetarian
  • Deficiencies — iron, vitamin D, B12, calcium, etc.
  • Health Conditions — diabetes, PCOS, thyroid, cholesterol
  • Supplements — iron tablets, multivitamins, vitamin D3
  • Health Reports — upload lab reports (.txt/.csv) for AI context

What AI Analyzes

  • Meal Composition — foods, drinks, supplements per meal
  • Nutrient Interactions — tea + iron, calcium + iron timing conflicts
  • Day Timeline — cross-meal pattern detection across all meals
  • Risk Assessment — personalized flags based on user profile
  • RAG-Enhanced Suggestions — curated nutrition knowledge base
  • Safety Guardrails — no diagnoses, no dosage changes, medical disclaimers

🔑 Multi-Key Rotation
Gemini client supports comma-separated API keys. On 429/quota errors, automatically rotates to the next key before falling back to OpenAI.
📊 Fallback Tracking
Every provider switch is tracked via MetricEvents — admin dashboard shows gemini_fallback, openai_answer, rule_fallback counts per agent.
Knowledge Retrieval

RAG System

Local keyword & tag-scored retriever enriches the Health Risk Agent with curated nutrition knowledge before LLM generation.

How Retrieval Works

Token Overlap Scoring
Matches query tokens against knowledge base content
Goal Match Boost (+5)
Higher score for docs matching user's health goals
Diet Type Match (+4)
Prioritizes diet-appropriate guidance
Condition Match (+5)
Boosts docs relevant to user's health conditions
Ayurveda Filter (+2)
Includes Ayurveda content only when relevant tags match
Top-6 Results
Returns ID, topic, content, source, evidence level, safety

Knowledge Base

📚 nutrition_goals.json
Curated nutrition guidance organized by goals (fat loss, muscle gain, deficiency reduction), diet types, and health conditions. Each entry has evidence level and safety tags.
🍃 ayurveda_food_combinations.json
Traditional Ayurvedic food combination guidance (e.g., milk + fruit). Labeled as traditional/cultural context — never presented as clinical advice.
🛡️ Safety Labels
Every knowledge item carries evidence_level and safety metadata. Ayurveda items are explicitly labeled as optional traditional guidance.

Tracing

LangSmith Integration

Optional LangSmith tracing for deep observability into the AI pipeline — with built-in PII protection.

What's Traced

  • Agent latency — duration_ms per agent stage (Meal Analyzer, Health Risk, Report)
  • LLM provider status — which provider handled each call (Gemini/OpenAI/rule)
  • Token usage — input/output tokens per call, estimated cost
  • API request metrics — method, path, status_code, duration_ms for every request
  • Trace IDs — appear in backend logs, Service Bus payload, and orchestrator logs for end-to-end correlation

PII Protection

🔒 Health Report Redaction
health_report_text is automatically redacted before sending to LangSmith to protect sensitive medical data.
✂️ Content Truncation
Long strings are truncated to 1200 characters in traces to reduce costs and protect privacy.
📊 Stored Locally Too
All metrics are also stored in the PostgreSQL metric_events table — no external dependency required for the admin dashboard.

Lessons Learned

Production Challenges

⚡ Gemini Free-Tier Daily Limit
Fix: 3-tier provider chain fallback (Gemini → OpenAI → rules). Multi-key rotation for quota management. All fallbacks tracked via MetricEvents.
🔒 Service Bus Message Lock Expiry
Fix: Added AutoLockRenewer with configurable lock renewal (300s default). Prevents message reprocessing on slow LLM calls.
🔑 Internal API Key Mismatch
Fix: Both backend and orchestrator require the same INTERNAL_API_KEY. Added to deployment checklist and env validation.
🌐 CORS Issues Across Environments
Fix: Dynamic CORS_ORIGINS config. Custom middleware handles edge cases. Different origins for local/ngrok/production.
🏗️ Frontend Build Wrong API URL
Fix: VITE_API_BASE_URL injected at build time via GitHub Secrets. Must be set correctly in CI/CD workflow.
📊 Observability Was Too Hard
Fix: Built a database-backed admin dashboard with 22+ metrics cards. No need to parse Azure logs — everything visible in the app.
🎯 RAGAS Results Not Visible
Fix: Added /internal/rag-eval-results endpoint. RAGAS eval script can publish results directly to dashboard with correct API key.
✅ Quick Debug Checklist
10-step production debugging runbook covering frontend URL, backend health, Service Bus queue, container status, trace IDs, and dashboard checks.
Design Patterns

Key Design Decisions

📮
Outbox Pattern
Meals are written to DB + outbox table atomically. Background publisher polls every 5s — reliable event delivery without distributed transactions.
🔄
Graceful Degradation
3-tier LLM fallback ensures the system never completely fails. Even without any API keys, rule-based analysis provides useful guidance.
📅
Day-Level Timeline
Full day's meal timeline sent to AI — enables cross-meal risk detection like tea 90 min after iron across different meals.
🏷️
Text Normalization
Free-text health conditions / supplements / deficiencies are normalized to canonical terms via alias dictionaries for consistent AI analysis.
📊
DB-Backed Metrics
All observability stored in MetricEvent table with JSON payloads — no external monitoring dependency. Admin dashboard queries directly.
🛡️
Safety-First AI
No medical diagnoses, no dosage changes, Ayurveda labeled as traditional guidance. Safety guardrails checked in evaluation with manual scoring rubric.
